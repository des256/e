# Habits

## Completing Work Before Moving On

Make sure you're actually 100% done with code. When you report code status, report the real status.

## Enforce Coding Standards

Make sure your coding standards are the same in the whole team. Use code formatter. Write a small MD file with coding standards.

## Document Chosen Patterns

Write a small MD file with the agreed libraries/patterns that are being used in the project. Include links to documentation of existing libraries/patterns. If the pattern is new, make separate MD file for it, containing when to use it, when not to use it, how to use it.

## Review New Patterns

Review new patterns before implementing them. Make some demonstration of the pattern, and show it to the rest of the team to ask for feedback. Get support.

## Never Expose Refactoring

Instead of making a ticket for refactoring (which bad managers can shoot down), come up with plan with the team to do incremental refactoring.

## Assume Unexpected Changes

Add extra upfront time for writing documentation.

## Scrum - it's all about confidence, not predicting the future

1. keep product owner out of the daily standup
2. put scrum master in their place
3. include extra time for new infrastructure, new tests, documentation, etc.
4. don't commit to multiple sprints
5. don't share the burndown chart with upper management
6. don't start working on a sprint without 100% acceptance criteria
7. deliver new cool features at every sprint
